## 二、本轮已完成的改动（全部测试通过）

### A. Key 级独立通道（ccLoad 风格渠道隔离）

**核心语义：key 通道的任何可重试失败（401/403/429/quota/5xx/超时/网络）只冷却该 key，绝不连坐 provider。**

1. `src-tauri/src/proxy/forwarder.rs`：
   - 重试主循环 `ErrorCategory::Retryable` 分支改为按 `key_id.is_some()` 分流（原来只有 `is_key_scoped_error` 的 401/429 类才走 key 路径，5xx/超时会 `provider_blocked_by_failure` 连坐整个 provider）。key 通道失败 → `release_channel_permit_neutral` + `record_key_failure_for_error` + 若同 provider 还有后续 key 则 `pending_key_retry_provider_id`（优先试同 provider 下一个 key）→ continue。
   - 三处整流调用点（media ~775 / signature ~939 / budget ~1098）同样分流：`if key_id.is_none() { blocked.insert } else if remaining_same_provider_key { pending_key_retry... }`；删除了 return Err 前的无效 insert。
   - `handle_rectifier_retry_failure`：is_provider_error 且 key 通道 → 记 key 冷却而非 `record_channel_result`。
   - `key_cooldown_seconds` 替换为 `key_failure_cooldown(error) -> (base, cap)`：401/403=10min→cap24h、429=1min→30min、quota=5min→1h、5xx=2min→30min、超时/网络=1min→30min。
   - `is_key_scoped_error` 保留（仅 NonRetryable 分支判定用）。
2. `src-tauri/src/database/dao/provider_keys.rs`：`record_provider_key_failure(app, provider, key, base_seconds, cap_seconds)` —— 冷却 = min(base × 2^已有连续失败, cap)（先 SELECT consecutive_failures 再算，移位 clamp 0..16）；base<=0 → Degraded 无冷却。成功（`record_provider_key_success`）清零恢复。
3. `src-tauri/src/proxy/provider_router.rs`：
   - `record_channel_result`：**仅 `key_id.is_none()` 时**调 `update_provider_health_with_threshold`（provider 健康只由无 key 池的通道驱动，key 失败不再写 provider_health）。
   - `record_key_failure` 签名透传 (base, cap)。
   - 调度排序 `apply_weighted_key_order` → `apply_key_order`：priority → consecutive_failures → last_used_at(LRU) → created_at → id。**weight 不再参与调度**（DB 字段保留兼容，ProviderKeyInput.weight 仍存在）。

### B. 智能默认故障转移（需求 B）

1. `src-tauri/src/database/dao/failover.rs` 新增 `get_default_failover_order(app_type)`：`SELECT id FROM providers WHERE app_type=? AND COALESCE(category,'')!='official' ORDER BY COALESCE(sort_index,999999), id`（官方登录卡不支持路由，排除——与前端「不支持路由」徽章判定 `category === "official"` 一致）。
